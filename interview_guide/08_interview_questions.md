# 面试高频问题

## 1. 这个项目和普通 ChatGPT 插件有什么区别？

普通插件更像“模型调用一个 API”。MiniDB Agent 是完整 Agent 系统，有任务理解、规划、工具选择、安全策略、状态管理、错误恢复和交付报告。模型不是唯一核心，工程化边界才是核心。

## 2. 为什么需要 Agent Loop？

数据库任务不是单轮问答。慢 SQL 诊断需要先查统计，再看表结构，再看执行计划，再给建议。Agent Loop 能把这些步骤串起来，并在每一步做权限判断和结果验证。

## 3. 为什么不直接让大模型生成 SQL？

因为数据库操作高风险。模型可以参与推理，但真正执行必须通过结构化工具、SQL 分类、安全策略和人工审批。否则容易误删数据、误改结构或泄露敏感信息。

## 4. 如何保证写操作安全？

写操作必须满足几个条件：工具允许、环境允许、SQL 分类通过、审批存在、审批 SQL hash 和当前 SQL 一致、目标环境一致。任何一个不满足都会被拦截。

## 5. 意图模块会不会让系统变得不灵活？

不会，因为意图不是枚举所有任务，而是抽取任务结构。比如“写测试报告”和“查慢 SQL”不需要硬编码每一种句式，而是抽取领域、目标、风险、输出要求和下一步动作。

## 6. 为什么参考 Codex 和 Claude Code？

它们代表了成熟终端智能体的两个关键方向：Codex 的执行过程可见、工具结果清晰、权限控制明确；Claude Code 的工作区会话、slash command 和协作体验成熟。MiniDB Agent 把这些思想迁移到 PostgreSQL 运维场景。

## 7. 如何处理模型工具调用格式不稳定？

在模型边界做规范化。模型如果返回原生 tool_calls，就直接用；如果返回 DSML/XML 工具调用文本，就解析成标准 tool_calls；如果消息历史违反 provider 协议，就在调用前清理。

## 8. 如何处理 SQL 错误？

读 SQL 错误可以给模型一次修复机会，比如字段名写错；重复同类错误就阻塞并生成错误报告。写 SQL 错误不会盲目重试，因为可能造成重复变更。

## 9. 如何验证项目真的可用？

不只跑单元测试，还要连接真实 PostgreSQL。验证流程包括：启动 CLI、配置数据库、查询慢 SQL、收集 top queries、生成报告、检查 delivery package 状态和 report 内容。

## 10. 这个项目最大的难点是什么？

难点不是调用模型，而是把不可控的模型输出放进可控的工程系统里。尤其是工具权限、状态一致性、错误恢复、CLI 体验和数据库安全。

## 11. 如果上线到生产环境怎么改？

production 默认只读；写操作不自动执行；审批接企业工单系统；审计日志落库；敏感字段脱敏；所有工具加超时和最大返回行数；高风险操作需要多人审批。

## 12. 后续如何扩展？

可以扩展 MySQL、Redis、ClickHouse 等数据源；接入企业 CMDB 和权限系统；增加慢 SQL 自动分类；做索引效果评估；把报告接入工单和知识库。
