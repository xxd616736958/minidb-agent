# 需求如何确定

## 起点问题

最初的目标不是做一个通用聊天助手，而是把 mini-agent 优化成“可以管理 PostgreSQL 数据库的智能体”。这意味着系统必须解决真实数据库场景里的问题：

- 用户不会总是说清楚环境、库名、表名、指标。
- 数据库操作有风险，不能让模型直接执行写 SQL。
- 慢 SQL 诊断需要访问系统视图、执行计划、索引和锁信息。
- 运维工作需要过程记录和交付报告。
- CLI 必须像 Codex/Claude Code 一样易用，不能要求用户先手动启动服务再配置一堆参数。

## 需求拆分方法

我把需求拆成三类。

第一类是用户体验需求：在目标目录启动、自动识别工作区、首次配置数据库、后续可复用配置、slash command 修改配置、实时展示工具调用和结果、支持会话恢复。

第二类是智能体能力需求：理解任务、规划步骤、选择工具、根据结果继续推理、失败后修复、最终交付。

第三类是数据库安全需求：默认只读、写操作审批、环境敏感、SQL 分类、审批绑定、验证和审计。

## 为什么参考 Codex 和 Claude Code

Codex 的启发主要是任务执行体验：它会展示当前执行的命令、结果摘要、失败信息和下一步，而不是把所有内部状态暴露给用户。

Claude Code 的启发主要是协作和会话体验：用户在工作目录里直接启动，通过自然语言和 slash command 协作，系统维护会话上下文和工具权限。

MiniDB Agent 最终不是照搬它们的代码结构，而是复用它们背后的设计原则：明确工作区、明确工具边界、明确权限请求、明确执行结果。

## 关键需求决策

- 不做通用助手，聚焦 PostgreSQL。
- 不把所有信息都向用户澄清，能从配置和数据库自己查到的就自己查。
- 不让模型直接写 SQL 执行，所有写操作走工具和审批。
- 不把工具错误隐藏，必须展示、记录、恢复或明确阻塞。
- 不只输出聊天回答，必须生成报告和审计证据。
