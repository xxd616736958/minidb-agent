# 企业中如何开发这个项目

## 1. 需求调研

先和 DBA、后端开发、SRE 讨论真实场景：

- 慢 SQL 排查要看哪些指标？
- 哪些操作必须审批？
- 哪些环境禁止写操作？
- 报告需要给谁看？
- 审计记录要保留哪些字段？

输出需求列表和风险边界。

## 2. 原型验证

先做最小闭环：

1. CLI 输入任务。
2. 连接 PostgreSQL。
3. 查询 `pg_stat_statements`。
4. 输出慢 SQL 列表。
5. 生成报告。

验证这个闭环可用后，再扩展审批、索引建议和错误恢复。

## 3. 工具优先

企业里不能让模型随便拼 SQL，所以应先建设工具层：

- 连接检查。
- 表结构查看。
- 慢 SQL 查询。
- EXPLAIN。
- 锁检查。
- 健康检查。
- 索引建议。
- 维护操作。

工具有了之后，模型只是选择和组合工具，而不是直接操作数据库。

## 4. 安全评审

上线前必须评审：

- production 是否默认只读。
- 写操作是否必须审批。
- 审批是否绑定 SQL hash。
- 日志是否脱敏。
- 错误时是否会自动重试危险操作。
- 报告是否泄露敏感数据。

## 5. 测试策略

测试分四层：

- 单元测试：SQL 分类、工具注册、状态路由。
- 集成测试：LangGraph 节点流转、审批恢复、错误恢复。
- CLI 测试：启动、会话、slash command、输出格式。
- 真实数据库验证：连接 PostgreSQL，完成慢 SQL 诊断和报告交付。

## 6. 灰度上线

先在 dev/test 环境只读开放，确认不会误操作；再开放 staging 的审批写操作；production 只保留只读诊断和报告，写操作需要更严格审批。

## 7. 监控与审计

企业落地需要记录：

- 用户请求。
- 目标环境。
- 工具调用。
- SQL hash。
- 审批记录。
- 执行结果。
- 报告路径。
- 错误和恢复动作。

这些信息用于复盘和合规。
